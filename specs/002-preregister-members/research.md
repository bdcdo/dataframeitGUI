# Research: Pré-registro de membros e vínculo de múltiplos e-mails

**Feature**: `002-preregister-members` | **Data**: 2026-06-11

Nenhum NEEDS CLARIFICATION restou na spec (3 clarificações resolvidas em sessão). Este documento registra as decisões técnicas que estruturam o desenho, com alternativas consideradas.

## D1 — Placeholder de pré-registro: Supabase-only, sem criar usuário no Clerk

**Decisão**: o pré-registro cria apenas `auth.users` + `profiles` no Supabase (via admin client), sem chamar `clerk.users.createUser()`. `addMember()` consulta o Clerk apenas para localizar o dono atual de um endereço já verificado; não cria conta no provedor.

**Rationale**: o fluxo anterior criava um usuário Clerk "fantasma" para cada e-mail desconhecido, poluía o tenant, já causou erro 422 `form_identifier_exists` em signups bloqueados por Turnstile e tornava a correção de e-mail uma operação em dois sistemas. O fluxo atual deixa o placeholder apenas no Supabase; quando a conta real aparece, `reconcileClerkUserAccess()` relê o estado atual do Clerk, encontra ou cria o mapping pelo e-mail primário verificado e conclui o auto-join exigido pelo FR-004.

**Alternativas consideradas**: (1) manter a criação de Clerk user (status quo) — rejeitada pelos problemas acima e porque o Clerk não envia convite nesse fluxo de qualquer forma (a spec dispensa e-mail); (2) usar Clerk Invitations API — rejeitada porque a spec decidiu explicitamente por pré-registro sem e-mail.

## D2 — Detecção de "pendente": coluna `profiles.activated_at`

**Decisão**: nova coluna `profiles.activated_at TIMESTAMPTZ NULL`. Pendente = `activated_at IS NULL`. É preenchida (uma única vez) por `reconcileClerkUserAccess`, sequência compartilhada entre os webhooks `user.created`/`user.updated` e a action idempotente `completeAccess()` quando a tela de conclusão repara explicitamente um vínculo incompleto. O vínculo administrativo de um e-mail pode guardar imediatamente o UUID de um profile pendente, mas não altera `activated_at`, pois não houve acesso autenticado. Na lista do projeto, um membro é apresentado como ativo quando seu profile canônico ou algum profile vinculado ativo comprova acesso naquele projeto; a ativação do alias não altera globalmente o profile canônico. `getAuthUser()` permanece read-only. Backfill da migration: `activated_at = created_at` para todos os profiles existentes.

**Rationale**: o sinal "tem conta real" não é derivável com segurança do estado atual — `clerk_user_mapping` existe também para placeholders antigos criados pelo fluxo vigente. Uma coluna explícita é barata, indexável e legível na query da lista de membros. O webhook cobre o fluxo normal; `completeAccess()` oferece reparo recuperável para contas pré-existentes ou eventual perda do webhook sem introduzir escrita no caminho de autenticação e renderização.

**Limitação aceita**: convidados antigos (criados pelo fluxo atual) que nunca acessaram serão backfillados como "ativos" — não há sinal local confiável para distingui-los; corrigir via Clerk `lastSignInAt` seria um script one-off fora do escopo.

**Alternativas consideradas**: (1) derivar pendência da ausência de `clerk_user_mapping` — falha para placeholders antigos que têm mapping; (2) consultar Clerk em runtime — chamada externa por render da lista, rejeitada por performance.

## D3 — Vínculo de e-mails: tabela única `member_email_links` com resolução por alias de projeto

**Decisão**: uma tabela `member_email_links (project_id, member_user_id, email, linked_user_id NULL, …)` serve simultaneamente de registro do vínculo e de alias. `member_user_id` é o membro-alvo; `linked_user_id` é o profile conhecido para aquele e-mail e é preenchido imediatamente mesmo quando o profile ainda está pendente, ficando `NULL` somente enquanto nenhum profile existe. O schema garante uma única identidade canônica por `(linked_user_id, project_id)`, proíbe self-alias, exige que o target seja membro do mesmo projeto e impede que a conta vinculada também exista em `project_members` naquele projeto; por isso a resolução é sempre de um único salto para uma membership terminal. Inserts e updates de vínculos e inserts de memberships tomam o mesmo advisory lock global. O acesso "como o membro" tem dois componentes: a RLS deriva acesso, papel, flags e own rows da membership canônica, enquanto a aplicação usa `getProjectAccessContext(projectId, user)` para páginas/layouts e `resolveProjectMemberActor(projectId)` como porta única das mutations pessoais, devolvendo conta autenticada e `memberUserId` no mesmo resultado discriminado.

**Rationale**: o caso "e-mail já pertence a outra conta" impede mapear o Clerk user direto para o profile do membro (isso seria merge global, descartado pela clarificação). O alias por projeto preserva os dois profiles globais e restringe a equivalência ao projeto — exatamente o FR-013. Usar uma única mecânica (sempre criar/usar o profile próprio da conta + alias) evita dois caminhos divergentes para "e-mail sem conta" vs "e-mail com conta". `getProjectAccessContext()` torna a distinção explícita: `accountUserId` identifica a conta autenticada em autoria e auditoria; `memberUserId` identifica a membership canônica em papel, permissões e trabalho do projeto.

**Alternativas consideradas**: (1) mapear o Clerk user do segundo e-mail diretamente para o profile do membro (`clerk_user_mapping` N:1) — simples, mas é merge global: a conta nova enxergaria todos os projetos do membro e quebraria o FR-013 no caso de conta pré-existente; rejeitada para manter mecânica única. (2) Duplicar `project_members` para a conta vinculada — quebraria a unicidade da identidade (contagens por `respondent_id`, comparações N+ e sorteio passariam a ver dois membros).

## D4 — Unificação de membros: função Postgres transacional via RPC

**Decisão**: função SQL `unify_project_members(p_project_id, p_source_user_id, p_target_user_id, p_linked_user_id, p_link_email, p_acting_user_id)` SECURITY DEFINER, chamada via RPC pelo admin client num server action com checagem prévia de coordenador. Migra, no escopo do projeto, somente identidades de trabalho existentes no schema final: `assignments.user_id` (resolvendo colisões do `UNIQUE(document_id, user_id, type)` — a do alvo prevalece, a duplicada do source é removida), `responses.respondent_id` (recalculando `is_latest` após a fusão), `reviews.reviewer_id`, `verdict_acknowledgments.respondent_id`, `field_reviews.self_reviewer_id/arbitrator_id` e `response_equivalences.reviewer_id`; remove a preferência do source em `researcher_field_orders`, deleta sua linha de `project_members` e grava o alias source→target com o dono e o e-mail revalidados. Campos de ator/auditoria (`author_id`, `resolved_by`, `created_by`) permanecem ligados à conta que realizou o ato. Source e target precisam ser memberships terminais: a conta já vinculada não chega à RPC como source. A função toma o advisory lock global de gestão de identidade, bloqueia as duas memberships em ordem determinística e fecha com `SHARE ROW EXCLUSIVE` as sete tabelas de trabalho lidas ou migradas. Arbitragem pendente entre source e target, colisão de reviews no mesmo `(document_id, field_name)`, conflito entre codificador e revisor de comparação ou colisão inesperada do e-mail principal abortam a transação antes da primeira mutação. O preview é uma RPC agregada que calcula as cinco contagens de impacto e conflito sem trazer datasets ilimitados ao servidor; a execução chama a mesma função novamente sob os locks. A tabela e as RPCs de atribuição tornam autoarbitragem e autocomparação pendentes irrepresentáveis; histórico concluído pode convergir para a identidade canônica quando não viola as constraints finais.

**Rationale**: são 10+ tabelas que precisam mudar atomicamente; server actions via PostgREST não têm transação multi-statement — uma função Postgres dá atomicidade e mantém a lógica de colisão num lugar só. A unificação é permanente (clarificação Q1), então não há estado de undo a persistir.

**Alternativas consideradas**: (1) sequência de UPDATEs no server action — sem atomicidade, estado parcial em caso de falha no meio; rejeitada. (2) View de "identidade efetiva" sem migrar dados — manteria dois respondent_ids para sempre e contaminaria todas as contagens (comparações, sorteio, progresso) com resolução de alias em cada query; rejeitada por complexidade difusa.

## D5 — Interação com a feature 001 (sorteio): nenhuma mudança necessária

**Decisão**: o sorteio (atual e o planejado em `specs/001-improve-assignment-lottery/`) seleciona o pool a partir de `project_members` — membros pendentes são linhas normais dessa tabela, logo são elegíveis automaticamente (FR-003) sem alteração no algoritmo; contas-alias não podem aparecer nessa tabela por invariante do schema. Único toque: exibir o badge "pendente" nas listas de seleção de pesquisadores do sorteio, para o coordenador saber quem ainda não acessou.

**Rationale**: o pré-registro foi desenhado de propósito como "membro comum + flag de ativação" para que todo o downstream (sorteio, atribuições, progresso) funcione sem casos especiais.

## D6 — Remoção de membro pendente: liberar atribuições pendentes

**Decisão**: `removeMember(memberId)` chama a RPC RLS `remove_project_member`, que deriva projeto e usuário da linha removível e deleta na mesma transação as `assignments` com `status = 'pendente'`. Os aliases do membro somem pelo `ON DELETE CASCADE` da FK composta quando a membership é removida; a RPC não duplica esse `DELETE`. Vale para qualquer membro, não só pendentes; documentos voltam ao conjunto "não atribuído" por consequência, sem janela de estado parcial entre a remoção e a limpeza.

**Rationale**: FR-005 exige que as atribuições do pendente removido retornem ao pool; membros pendentes só podem ter atribuições `pendente` (nunca acessaram, não iniciaram trabalho), então deletá-las é seguro e idempotente.

## D7 — Reconciliação Clerk: identidade verificada e conclusão explícita

**Decisão**: o e-mail primário verificado, localizado pelo ID primário do Clerk, define qual profile Supabase será criado ou recuperado; todos os e-mails verificados da mesma conta podem resolver aliases. O mapping persiste `access_sync_version`, `access_snapshot_version` e `clerk_deleted`. Cada reconciliação usa `user.updatedAt` como geração e executa duas transações: `begin_clerk_access_snapshot` escolhe a geração e grava `access_sync_version = 0`; `complete_clerk_access_snapshot` só aceita a mesma geração e aplica profile, lista completa de aliases e marker `1` atomicamente. Se a geração foi superada, o código relê o Clerk uma vez e tenta somente o snapshot atual. `publicMetadata.supabase_uid` é o último efeito e só é regravado quando diverge do UUID observado. `resolveAuth` aceita a sessão apenas quando metadata, mapping e marker concluído existem e concordam.

**Rationale**: selecionar `email_addresses[0]`, aceitar endereço não verificado ou publicar metadata antes dos efeitos locais criava estados ambíguos que o caminho de leitura teria de reparar. A primeira fase commitada derruba o acesso antes de qualquer efeito que possa falhar; a segunda mantém profile, aliases e marker final atômicos. Um Clerk sem primário verificado aplica snapshot com aliases vazios e sem ativação, mantendo o marker em `0`; sem mapping, não cria identidade. `user.deleted` e uma releitura 404 usam `begin_clerk_user_revocation` para gravar marker `0` e `clerk_deleted = true`, seguidos de `complete_clerk_user_revocation` para remover aliases. Assim, token ou evento antigo não restaura acesso, e `completeAccess()` oferece retry idempotente sem escrever no render autenticado.

## D8 — Lookup administrativo não prova a identidade dona do e-mail

**Decisão**: `linkMemberEmail` mantém dois conceitos separados. `profileByEmail` é o profile localizado por `profiles.email`; ele representa o pré-registro ou dado administrativo associado ao texto digitado. `ownerProfile` só existe depois que `reconcileVerifiedClerkEmailOwner()` confirma a posse atual no Clerk, relê a conta e devolve o UUID mapeado a partir do mesmo snapshot verificado. Se ambos existem e têm UUIDs distintos, são candidatos separados no preview; a action nunca escolhe um deles por coincidência de string. Depois da reconciliação, a action relê membership, vínculo e `profileByEmail` antes de decidir, porque os aliases podem ter convergido no intervalo.

**Rationale**: `profiles.email` não é uma prova atual de posse: pode ser placeholder, dado legado ou endereço que a conta já removeu. Fazer dele um fallback de identidade concederia acesso a um dono não confirmado e forçaria guards defensivos posteriores. A distinção torna o estado ruim irrepresentável no contrato: lookup administrativo informa o fluxo; somente `ownerProfile` confirmado pelo Clerk torna o acesso `ready`.
